# 繼續談信實與愛

## 發想 由RPA 舉例說明

其實，看我們的程式設計，就可以看到信實與愛的重要性了，而且，我們也可以看到 `信` 所扮演的重要角色。
信是依賴於 `實` 的，沒有 `實` ，`信` 就會因為基於虛謊，終會因為與事實不符而崩塌，然後，信任自然被破壞，每個個體只好築出各式各樣的防護機制，來保護自己，結果就是整個系統變得很笨重、難維護，甚至會崩潰。

例如，RPA(Robotic Process Automation)就是一個例子。
像我先前在幫人設計RPA的時候，PM或者我會先了解客戶想要甚麼，然後，將他們的需求，變成一個個具體的步驟，然後，每個步驟都有他要達成的目標，取得要取得的資訊，或者做出該有的動作。若有任何例外，或者錯誤，都要老實的回報，好讓客戶知道現在的狀況，然後，可以做適當的處置。若問題是屬於我設計上的問題，或者客戶的需求本來就有問題，我會誠實的告知，然後，一起討論出最好的解決方案，然後，再把程式修改好，交付給客戶。這樣，客戶才會對我，對這個RPA有信心，然後，才會願意繼續使用，或者擴大使用範圍。
而這裡的步驟，比方說要由ERP軟體取得某些資訊，那麼，我會先將這個步驟變成幾個子步驟，比方說，1. 先打開ERP軟體，然後，2. 登入帳號密碼，3. 輸入查詢的字串，4. 取得資訊，5. 關閉ERP軟體。而這五個就是五個子步驟。
原則上，這五個子步驟都有不同的處理方式，比方說，打開ERP軟體，由於有時有一些特殊限制，不能直接從桌面打開，所以，需要模擬 `Win-R` 鍵來輸入命令等，所以，若發現 ERP打不開，我只要修這個子元件即可。
而子步驟2，要看我能否取得UI element的讀取入口，若沒有的話，可能只好用螢幕辨識來找到該欄位，然後，再輸入帳號密碼。有時會有中文輸入法的問題，導致輸入錯誤，所以，子步驟2就是負責處理登入帳號密碼的元件。而解決輸入法的問題，由於任意的輸入都有可能遇到，所以，這個部分就會提升到 Utils 的階層，讓所有的元件可以使用。
而像是子步驟4，取得資訊，最糟的是得靠 OCR 來處理，好一點的可以用 UI element 來取得，再好一點的，可以直接從資料庫取得，或者透過 API 取得。所以，在接一個案子的時候，有時候真的很難說，會不會有意外。能跑RPA是一回事，能否總是跑成功卻是另外一回事。
像是輸入的值使用者給的資料可能出錯，如把 `l` 寫成 `1`，怎麼確定使用者給的資料是否正確，也變成一個問題，甚至要變成一個步驟，安在流程取得資料後的後面做檢查。

由上面的例子我們可以看到，整個RPA流程的出現，乃是基於我們有一個目的，這是高於所有的元件之上，穩定而正確的完成客戶的需求就是它們存在的目的。若不能正常運作，整個RPA程式原則上就是一改再改，直到可以正常運作為止。站在我的立場，我是基於愛(為了系統能穩定持續的運作)來設計RPA的，而對RPA裡面的元件而言，若我設計的RPA很穩定，那他們也會是穩定的存在，不也可以說是一種`愛`嗎？不過，這裡也可以看到，若無法滿足實際，這愛就無法站立得住，勢必要修改。
此外，每個元件的設計，對我和對元件而言，也是基於信實。我乃實實在在地設計，善用intellisense 與 trace 來確保程式的穩定性，只要有任何錯誤或不穩，就是老實的面對，設法找到解決方法。最頭痛的是當所用的 packages 本身就很有問題，或者甚至是OS本身的特殊癖好，就會破壞我對他的信任。還好，通常上網常常可以找到解決的方法，不過，常常遇到的是OS的安全要求導致奇怪的錯誤。這時，就需要多做許多的程式碼來處理，互不信任，就是得多做這些動作。
對元件而言，設計時，我會盡量用強型別來避免因為資料型態的問題，導致程式出錯，然後，也會善用 AI 來幫我看看是否有隱憂，或者是否有更好的寫法。原則上，跑的時候，這些元件會被一個一個的執行，他們都按照我的設計，實實在在的跑他該做的，也信任與他相連的元件們，也是這樣的實實在在。其實，他們若實實在在的做他該做的，並且信實的將資訊傳給下一個元件，這其實也就是一種實現在小尺度裡的 `愛` 不是嗎？因為這樣，與他相連的元件，也才能因為正確的資料而正確的反應。

以前我在寫程式的時候，若發現有問題，比如型態出錯，我就是在出錯的地方加上檢查，可是，這樣下來，慢慢的程式變得非常胖，雜亂無章，難以維護。現在，我會使用像 ETL 的作法，將資料先整理好，再做處理。這樣一來，因為資料已經先經過清理，後面的程式只要信任資料的正確性，程式碼可以專注於邏輯本身，因此，就會變得比較清爽，也比較容易維護。

再者，`信` 的重要性，在這個RPA的例子我們還可以看到，每個步驟或元件，其實他們是各司其職。登入帳號密碼的子元件，只要能夠利用螢幕辨識找到輸入帳號密碼的欄位，並且輸入即可，他並不需要知道如何打開ERP，也不需要知道如何取得資訊。 知道如何打開ERP，是他的上一個子元件的任務，他只要信任他已經打開即可。而他們的父元件，只要能夠正確的根據這些子元件傳回的值來調度他們即可，他既不用知道怎麼開app，也無須知道怎麼跟OS溝通。
所以，他們只要互相信任，相信別人是實實在在的完成他們該做的事，不然，越上層的元件，就得知道越多的細節，整個程式就會變得越複雜，最後被自己要負責的細節所壓垮。所以，就算像 RPA 這種系統，也是需要建立在信實的基礎上，才能正常運作，更遑論複雜如人一般的生命體的存在，我們的細胞們，不也是這樣信實的連結在一起嗎？

所以，由這個小例子我們可以看到，`愛` 是我們與整個RPA程式運作的目的，然而，這得是基於`實`才能站立得住。然而，我們不可能讓每個子元件都處理到最底層，所以，我們還需要 `信`。所以，無論是設計程式的我，還是RPA元件們，我們都是基於 `信實` 來行，才能讓整個RPA程式正常運作，才能談到`愛`。

所以，沒有 `信實`，談的 `愛`，很容易就偏頗掉，對這個例子而言，就是得修改程式，好滿足 `信實` 的要求，

事實上，我們看整個軟體世界，不也都遵循這樣的原則嗎？就算大到像 Microsoft Azure 這樣的大型雲端平台，也是由無數的元件所組成，也都各司其職，才能提供這些豐富的服務，不是嗎？

如果，整個世界要能夠像是一個非常穩定的服務存在，不會一天到晚有爭戰、有衝突，不也應該要像這樣，基於 `信實` 與 `愛` 來運作嗎？
再次強調，`愛` 必須是基於 `信實` 才能站立得住，否則這個愛，很容易只是讓小部分人得利，而大部分人受苦。
